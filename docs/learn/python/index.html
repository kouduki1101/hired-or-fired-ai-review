Python Review Pattern Book

AIコードを見抜くためのPythonレビュー型帳

これはPythonを書くための教科書ではありません。AIが生成したコードを見た瞬間に、 「これはどのカテゴリの不具合か」「どこが間違っているか」「正しい修正は何か」が 頭に浮かぶようにするための、正解付きレビュー筋トレ表です。

Instant Review Checklist

  • まずカテゴリを決める。構文ではなく、どの種類の事故かを見る。
  • 怪しい行を1行で言語化する。どこがズレているかを曖昧にしない。
  • 仕様の言葉とコードの演算子・条件・戻り値を照合する。
  • 正解コードを頭に浮かべる。書けなくても、修正の方向は言える状態にする。
  • 最後に、同じ型のバグが他の行にもないか横展開する。

1. カテゴリを先に見る

境界値、ロジック、データフロー、権限、要件読み違い。まず事故の種類を決める。

2. 怪しい箇所を短く指摘する

`if not value`、`<= 18`、`break`、`except Exception` のように、レビューで刺す場所を決める。

3. 正解を先に見て型にする

このページでは正解コードを隠さない。レビュー面接では、瞬間的に修正方針が浮かぶことが目的。

Basic境界値・入力検証対応問題: プロフィール表示のNone/空文字レビュー

None と空文字を同じものとして扱っていないか

失敗パターン: truthy / falsy の混同

怪しいコード
def display_name(nickname):
    if not nickname:
        return "Anonymous"
    return nickname
正解として浮かべる修正
def display_name(nickname):
    if nickname is None:
        return "Anonymous"
    return nickname
どこが間違っているか

`if not nickname:` が None だけでなく空文字も拾っている。

なぜ危ないか
  • 仕様が「nickname が None のときだけ Anonymous」と言っているなら、空文字はユーザーが明示的に入れた値かもしれない。
  • `not value` は None、空文字、0、空リスト、空辞書をまとめて false 扱いする。
  • AI生成コードは短く書くために `if not ...` を使いがちだが、業務仕様では区別が必要なことが多い。
瞬間レビュー反射
  • `if not value` を見たら、None / 空文字 / 0 を同じ扱いにしてよいか確認する。
  • 仕様に「未設定の場合だけ」とあれば、まず `is None` を候補にする。
  • 画面表示・プロフィール・任意入力では、空文字が有効値かを疑う。
この型をすぐ試す

プロフィール表示のNone/空文字レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Basic境界値・入力検証対応問題: ユーザー登録条件の境界値レビュー

境界値ちょうどを落としていないか

失敗パターン: 以上 / 超過、未満 / 以下の取り違え

怪しいコード
def can_register(user):
    if user.get("age") <= 18:
        return False
    return True
正解として浮かべる修正
def can_register(user):
    if user.get("age") < 18:
        return False
    return True
どこが間違っているか

`<= 18` により、18歳ちょうども登録不可になっている。

なぜ危ないか
  • 仕様が「18歳以上は登録可能」なら、落としてよいのは 17 歳以下。
  • `<=` と `<` の1文字差で、ちょうど境界のユーザーだけが誤判定される。
  • 境界値バグはテストが 17 / 19 だけだと見逃されやすい。
瞬間レビュー反射
  • 「以上」「以下」「未満」「超過」が出たら、ちょうどその値を頭で実行する。
  • 年齢、在庫、点数、金額、日数は境界値レビューの最優先対象。
  • 自然言語の仕様と比較演算子を必ず1対1で照合する。
この型をすぐ試す

ユーザー登録条件の境界値レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Basicロジック間違い対応問題: 在庫集計ループのcontinue/breakレビュー

スキップしたいだけなのにループを止めていないか

失敗パターン: break / continue / return の取り違え

怪しいコード
def count_available(items):
    count = 0
    for item in items:
        if item["stock"] == 0:
            break
        count += item["stock"]
    return count
正解として浮かべる修正
def count_available(items):
    count = 0
    for item in items:
        if item["stock"] == 0:
            continue
        count += item["stock"]
    return count
どこが間違っているか

`break` により、在庫0の商品を見つけた瞬間に後続の商品をすべて捨てている。

なぜ危ないか
  • 在庫0の商品を集計対象から外したいだけなら、ループ全体を終了してはいけない。
  • `break` は以降のデータを処理しない。`continue` はその1件だけを飛ばす。
  • AI生成コードは「除外条件」を書くときに、止めるのか飛ばすのかを曖昧にしやすい。
瞬間レビュー反射
  • `break` を見たら「本当に後続データを捨ててよいか」と問う。
  • `return` がループ内にあれば、最初の1件だけで終わっていないか疑う。
  • 除外・無視・スキップという仕様なら、まず `continue` を候補にする。
この型をすぐ試す

在庫集計ループのcontinue/breakレビュー で、カテゴリ選択から正解コード選択まで練習できます。

Basicロジック間違い対応問題: 注文割引と権限チェックの複合レビュー

件数・数量・率・固定額を混同していないか

失敗パターン: 集計単位と計算単位のズレ

怪しいコード
def apply_discount(total):
    if total >= 10000:
        total = total - 20
    return total
正解として浮かべる修正
def apply_discount(total):
    if total >= 10000:
        total = total * 0.8
    return total
どこが間違っているか

`total - 20` が、20%割引ではなく20円引きになっている。

なぜ危ないか
  • 仕様が20%割引なら、金額から固定値20を引くのではなく、0.8倍にする必要がある。
  • 同じ `total` でも、件数、数量、金額、比率のどれを表すかで正しい式が変わる。
  • 単位を読み違えると、コードは動くのに業務結果だけが静かに壊れる。
瞬間レビュー反射
  • 割引、税率、手数料、在庫集計では、単位を声に出して読む。
  • `+= 1` は件数、`+= quantity` は数量。どちらが仕様か確認する。
  • 「%」が仕様にあるのに固定値を足し引きしていたら赤信号。
この型をすぐ試す

注文割引と権限チェックの複合レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Intermediateデータフロー不整合対応問題: mutable defaultとコピーの参照レビュー

前回呼び出しの状態が残っていないか

失敗パターン: mutable default argument

怪しいコード
def add_tag(name, tags=[]):
    tags.append(name)
    return tags
正解として浮かべる修正
def add_tag(name, tags=None):
    if tags is None:
        tags = []
    tags.append(name)
    return tags
どこが間違っているか

`tags=[]` が関数定義時に一度だけ作られ、呼び出し間で共有される。

なぜ危ないか
  • Pythonのデフォルト引数は呼び出しごとではなく、関数定義時に評価される。
  • 1回目の `add_tag` で追加した値が、2回目の呼び出しにも残る。
  • リストや辞書をデフォルト引数にしたコードは、AI生成でも人間のコードでも頻出のレビュー対象。
瞬間レビュー反射
  • 引数に `=[]` や `={}` を見たら、ほぼ反射で疑う。
  • 呼び出しごとに独立すべき状態なら、デフォルトは `None` にする。
  • 共有したい状態なのか、偶然共有されているだけなのかを切り分ける。
この型をすぐ試す

mutable defaultとコピーの参照レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Intermediateデータフロー不整合対応問題: mutable defaultとコピーの参照レビュー

コピーしたつもりで元データを壊していないか

失敗パターン: 参照とコピーの混同

怪しいコード
def mask_user(user):
    masked = user
    masked["email"] = "***"
    return masked
正解として浮かべる修正
def mask_user(user):
    masked = user.copy()
    masked["email"] = "***"
    return masked
どこが間違っているか

`masked = user` はコピーではなく、同じ辞書への別名を作っている。

なぜ危ないか
  • `masked` を変更すると、元の `user` も変更される。
  • マスキング、整形、表示用変換では、元データを破壊してはいけないことが多い。
  • 代入はコピーではない、という基本を忘れると、後続処理で原因不明のデータ汚染が起きる。
瞬間レビュー反射
  • `copy = original` を見たら、名前だけコピーしていないか疑う。
  • 入力を破壊してよい関数か、表示用の別オブジェクトを返す関数かを確認する。
  • ネストがある場合は shallow copy で足りるかも追加で見る。
この型をすぐ試す

mutable defaultとコピーの参照レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Intermediate要件の読み違い対応問題: 例外処理と戻り値契約のレビュー

失敗を成功レスポンスに変換していないか

失敗パターン: 例外処理と戻り値契約の破壊

怪しいコード
def parse_amount(text):
    try:
        return {"ok": True, "value": int(text)}
    except Exception:
        return {"ok": True, "value": 0}
正解として浮かべる修正
def parse_amount(text):
    try:
        return {"ok": True, "value": int(text)}
    except ValueError:
        return {"ok": False, "value": None}
どこが間違っているか

`except Exception` の中で `ok: True` を返しており、失敗が成功に見える。

なぜ危ないか
  • 呼び出し側は `ok` を見て成功だと判断するため、入力エラーに気づけない。
  • `except Exception` は範囲が広く、想定外のバグまで握りつぶす。
  • エラーを値で返すなら、成功時と失敗時の契約を明確に分ける必要がある。
瞬間レビュー反射
  • `except Exception` を見たら、握りつぶしているバグがないか確認する。
  • エラー時に `ok=True`、空配列、0などを返していないか疑う。
  • 戻り値契約は、成功と失敗が呼び出し側から区別できる形かを見る。
この型をすぐ試す

例外処理と戻り値契約のレビュー で、カテゴリ選択から正解コード選択まで練習できます。

Intermediateデータフロー不整合対応問題: generator消費とcomprehension条件レビュー

generator を二回使って空にしていないか

失敗パターン: generator 消費と内包表記条件のズレ

怪しいコード
def summarize_scores(scores):
    passing = (score for score in scores if score >= 70)
    count = len(list(passing))
    passing_avg = sum(passing) / count
    failed = [score for score in scores if score >= 70]
    return passing_avg, failed
正解として浮かべる修正
def summarize_scores(scores):
    passing = [score for score in scores if score >= 70]
    count = len(passing)
    passing_avg = None if count == 0 else sum(passing) / count
    failed = [score for score in scores if score < 70]
    return passing_avg, failed
どこが間違っているか

`len(list(passing))` で generator を消費した後に、同じ `passing` を `sum` している。

なぜ危ないか
  • generator は一度読み切ると空になるため、`sum(passing)` は期待値にならない。
  • `failed` の条件も `>= 70` になっており、不合格ではなく合格点を集めている。
  • 内包表記は短く見えるぶん、条件が逆でも流し読みしやすい。
瞬間レビュー反射
  • `list(generator)` の後に同じ generator を使っていたら消費済みを疑う。
  • passing / failed、include / exclude の条件は対になっているか見る。
  • 平均計算では0件時のゼロ除算も同時に確認する。
この型をすぐ試す

generator消費とcomprehension条件レビュー で、カテゴリ選択から正解コード選択まで練習できます。

Advancedデータフロー不整合対応問題: async/awaitと認可分岐のレビュー

非同期APIの結果を await せずに使っていないか

失敗パターン: await 漏れ

怪しいコード
async def load_dashboard(user_id, client):
    profile = client.get_profile(user_id)
    orders = await client.get_orders(user_id)
    if profile["disabled"]:
        raise PermissionError("disabled")
    return {"profile": profile, "orders": orders}
正解として浮かべる修正
async def load_dashboard(user_id, client):
    profile = await client.get_profile(user_id)
    orders = await client.get_orders(user_id)
    if profile["disabled"]:
        raise PermissionError("disabled")
    return {"profile": profile, "orders": orders}
どこが間違っているか

`profile = client.get_profile(user_id)` に `await` がなく、profile が実データではなく coroutine になっている。

なぜ危ないか
  • coroutine object は辞書ではないため、`profile['disabled']` で壊れる。
  • 片方のAPIだけ await されているコードは、レビューで見落としやすい。
  • 認可判定は、非同期取得が完了した実データに対して行う必要がある。
瞬間レビュー反射
  • async関数内で外部API呼び出しを見たら、返り値を使う前に await されているか確認する。
  • 同じ client の片方だけ await されている場合は、もう片方を疑う。
  • await漏れは型が緩い環境ほどレビューで拾う必要がある。
この型をすぐ試す

async/awaitと認可分岐のレビュー で、カテゴリ選択から正解コード選択まで練習できます。

Advanced権限・セキュリティ対応問題: キャッシュと認可順序のレビュー

キャッシュが認可を迂回していないか

失敗パターン: cache before authorization / 粗すぎる cache key

怪しいコード
CACHE = {}

def get_report(user, report_id, db):
    key = report_id
    if key in CACHE:
        return CACHE[key]
    if not user["can_view_reports"]:
        raise PermissionError("forbidden")
    report = db.load_report(user["tenant_id"], report_id)
    CACHE[key] = report
    return report
正解として浮かべる修正
CACHE = {}

def get_report(user, report_id, db):
    if not user["can_view_reports"]:
        raise PermissionError("forbidden")
    key = (user["tenant_id"], report_id)
    if key in CACHE:
        return CACHE[key]
    report = db.load_report(user["tenant_id"], report_id)
    CACHE[key] = report
    return report
どこが間違っているか

キャッシュ確認が認可より先にあり、さらに key が `report_id` だけで tenant を分離できていない。

なぜ危ないか
  • 権限のないユーザーでも、キャッシュ済みならレポートを取得できる。
  • 別テナントで同じ report_id がある場合、他社データを返す可能性がある。
  • 高速化のコードは、セキュリティ境界をすり抜ける典型的な場所。
瞬間レビュー反射
  • キャッシュより前に認可があるかを必ず見る。
  • マルチテナントでは cache key に tenant_id が含まれているか確認する。
  • 高速化・共通化・メモ化は、権限境界を壊していないか疑う。
この型をすぐ試す

キャッシュと認可順序のレビュー で、カテゴリ選択から正解コード選択まで練習できます。
