# 02. 非機能要件定義書

- 版数: 1.0 (Draft for Review)
- 適用範囲: AIOS Control Plane(SaaS / Self-Hosted 共通。差分は明記)

---

## NFR-PF: 性能

| ID | 要件 | 目標値 |
|---|---|---|
| NFR-PF-01 | タスクルーティング判断のオーバーヘッド(受付→スロット決定) | p95 < 50ms、p99 < 120ms(推論自体の時間は含まない) |
| NFR-PF-02 | 制御サイクル(指標算出→判定→作用発行)の完了時間 | K=100スロットで < 60秒(プローブ実行を含む場合 < サイクル周期の50%) |
| NFR-PF-03 | リネージ照会API(FR-GV-02) | 過去1年分の履歴に対し p95 < 3秒 |
| NFR-PF-04 | ダッシュボード初期描画 / 指標更新反映 | < 2秒 / < 10秒(SSE/WebSocket) |
| NFR-PF-05 | イベント書込みスループット | テナントあたり 1,000 events/sec を取りこぼしなく永続化 |
| NFR-PF-06 | Rehatch実行時間 | Prompt-Recompose/TV-Init: < 30秒。Distillation等の学習系: 非同期ジョブ(進捗可視化必須) |

## NFR-SC: スケーラビリティ

| ID | 要件 | 目標値 |
|---|---|---|
| NFR-SC-01 | スロット数 | 1コホートあたり K ≤ 1,000、1テナントあたり合計 ≤ 10,000 |
| NFR-SC-02 | テナント数(SaaS) | 単一クラスタで 1,000テナント。水平分割(セル)で以降拡張 |
| NFR-SC-03 | 教師ベクトル次元 | ≤ 4,096次元。次元拡張操作は無停止で完了 |
| NFR-SC-04 | 履歴保持 | イベント無期限(コールド階層へ自動移行)。ホット照会対象は直近13ヶ月 |
| NFR-SC-05 | ワーカー水平スケール | 指標演算・Rehatch・プローブ実行ワーカーはステートレスで水平スケール可能 |

## NFR-AV: 可用性・信頼性

| ID | 要件 | 目標値 |
|---|---|---|
| NFR-AV-01 | SaaS 稼働率(タスクルーティングAPI) | 99.9%/月(Enterprise SLA: 99.95%) |
| NFR-AV-02 | 制御ループ停止時の縮退 | ルーティングは最終既知の指標で継続動作(制御と推論経路の分離) |
| NFR-AV-03 | RPO / RTO | RPO ≤ 5分(イベントストア) / RTO ≤ 1時間 |
| NFR-AV-04 | Rehatch失敗時 | 自動ロールバック(直前世代スナップショット)。ロールバック不能時はスロットを `DORMANT` にしアラート |
| NFR-AV-05 | 外部LLM/Embedding API障害時 | リトライ(指数バックオフ)+サーキットブレーカ。プローブ欠測サイクルは指標更新をスキップし欠測を記録(EMAを汚さない) |
| NFR-AV-06 | 冪等性 | タスク投入・Rehatch指示・承認APIは冪等キー対応 |

## NFR-SE: セキュリティ

| ID | 要件 |
|---|---|
| NFR-SE-01 | 保存時暗号化(AES-256、テナント別鍵/KMS)、転送時TLS1.2+ |
| NFR-SE-02 | テナント分離: 行レベル分離+アプリケーション層での二重チェック。Enterprise向けにDB物理分離オプション |
| NFR-SE-03 | RBAC(FR-TN-02)と全管理操作の監査ログ(操作者・理由・前後値) |
| NFR-SE-04 | シークレット管理: 顧客のLLM APIキー等はVault/KMS管理、ログ・イベントへの混入防止(マスキング) |
| NFR-SE-05 | プロンプトインジェクション対策: タスク入力は制御プレーンの判断(閾値・設定)に影響を与えないよう境界分離。禁止ベクトル監視(FR-SF)と併用 |
| NFR-SE-06 | 脆弱性管理: 依存関係SCA・コンテナスキャンをCIで必須化。重大脆弱性は14日以内に修正リリース |
| NFR-SE-07 | ペネトレーションテスト: GA前および年1回の第三者診断 |

## NFR-CP: コンプライアンス・監査適合

| ID | 要件 |
|---|---|
| NFR-CP-01 | EU AI Act等の透明性・記録保持要件に対応可能なリネージ/監査エクスポート(FR-GV-01〜03)。記録は改竄検知可能(ハッシュチェーン+定期アンカー) |
| NFR-CP-02 | ISO/IEC 42001(AIマネジメントシステム)の運用記録要件にマッピングした監査レポートテンプレートを提供 |
| NFR-CP-03 | 個人データの取扱い: タスク本文の保存要否をテナントが選択可能(既定: 埋め込みのみ保存し原文はTTL付き)。GDPR消去要請には原文削除+プロセス正当性説明(¶0227)で対応 |
| NFR-CP-04 | データレジデンシ: SaaSはリージョン選択(JP/EU/US)。Self-Hostedは顧客環境完結 |
| NFR-CP-05 | 特許表示: 製品UI・ドキュメントに特許番号を表示(virtual patent marking) |

## NFR-OP: 運用性・可観測性

| ID | 要件 |
|---|---|
| NFR-OP-01 | 全コンポーネントが構造化ログ(JSON)+OpenTelemetry Trace/Metricsを出力。制御サイクルはトレースIDで一気通貫追跡可能 |
| NFR-OP-02 | ヘルスチェック(liveness/readiness)、グレースフルシャットダウン(実行中Rehatchの安全完了または安全中断) |
| NFR-OP-03 | 設定はコード化(宣言的YAML/API)。閾値・α・β等の変更はバージョン管理され即時ロールバック可能 |
| NFR-OP-04 | Self-Hosted: Docker Compose(評価用)とHelm Chart(本番)を提供。エアギャップ環境向けオフラインバンドル(Enterprise) |
| NFR-OP-05 | ゼロダウンタイムアップグレード(DBマイグレーションは後方互換2バージョン維持) |
| NFR-OP-06 | バックアップ: DB日次+WAL連続、オブジェクトストレージのバージョニング。リストア手順は四半期ごとに演習 |

## NFR-MT: 保守性・拡張性

| ID | 要件 |
|---|---|
| NFR-MT-01 | 指標アルゴリズム(散逸度①〜⑤、適合度)・Rehatch戦略・ルーティングポリシーはプラグイン機構で追加可能(コア改変不要) |
| NFR-MT-02 | ドメイン層(指標演算・選定ロジック)は純関数として実装し、ユニットテストカバレッジ ≥ 90% |
| NFR-MT-03 | 公開APIはOpenAPI 3.1で定義しSDKを自動生成。破壊的変更はメジャーバージョンのみ、12ヶ月の旧版並行稼働 |
| NFR-MT-04 | Adapter適合テストキット(サードパーティ実装の互換性検証)を提供 |

## NFR-UX: 使用性

| ID | 要件 |
|---|---|
| NFR-UX-01 | ダッシュボードは日本語・英語対応(i18n基盤)。タイムゾーン対応 |
| NFR-UX-02 | 主要ブラウザ最新2バージョン対応、WCAG 2.1 AA準拠(色のみに依存しない健全性表示 — FIXED/STABLE/CHAOTICは形状+ラベル併用) |
| NFR-UX-03 | オンボーディング: サンプルコホート+チュートリアルで、初回起動から群の可視化まで30分以内 |

## NFR-CT: コスト制御

| ID | 要件 |
|---|---|
| NFR-CT-01 | プローブ実行・埋め込み算出のLLM/Embedding APIコストをテナント別に計測し、予算上限(超過時はサンプリング率自動低減)を設定可能 |
| NFR-CT-02 | 指標演算は増分計算(全再計算はキャリブレーション時のみ)。散逸度①(出力埋め込み方式)を既定とし計算負荷を最小化(¶0209) |

## 容量計画の目安(参考値)

| 項目 | 想定 |
|---|---|
| イベントサイズ | 平均2KB(埋め込み参照は別テーブル) |
| K=100・5分サイクル・タスク1万/日のテナント | イベント約 300万/月 ≒ 6GB/月(ホット) |
| 埋め込み(1536次元 float32) | 6KB/件。プローブ20問×K=100×288サイクル/日 ≒ 3.5GB/日 → サンプリングと世代集約で1/10に圧縮 |
